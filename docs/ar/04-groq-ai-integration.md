# تكامل Groq AI

هذه الوحدة تتصل بـ API المجاني لـ Groq لتوليد أوصاف العمليات باستخدام نموذج لغة كبير. التكامل مصمم ليكون **مرناً، محدود المعدل، وبدون إعداد** للمستخدم النهائي.

## لماذا Groq؟

| العامل | Groq | Gemini (جرّب) | OpenAI (جرّب) | LLM محلي (اعتُبر) |
|--------|------|----------------|----------------|------------------------|
| المستوى المجاني | 14,400 طلب/يوم | 1,500 طلب/يوم | أرصدة منتهية | 0 (لكن يتطلب 8GB+ ذاكرة) |
| بدون بطاقة ائتمان | ✅ | ✅ | ❌ | ✅ |
| السرعة | 300+ رمز/ث (LPU) | متوسطة | سريعة | بطيئة على Mac Intel |
| متوافق مع OpenAI | ✅ | ❌ | ✅ | يختلف |

معالج LPU المخصص من Groq يجعله **أسرع 10-50 مرة** من الاستدلال القائم على GPU للاستعلامات القصيرة، وهو مثالي لحالة الاستخدام لدينا (مئات الردود القصيرة بـ 30 رمزاً).

## الإعداد

### 1. يحصل المستخدم على مفتاح API مجاني

```
https://console.groq.com  →  سجّل (Google/GitHub)  →  أنشئ مفتاح API
```

لا بطاقة ائتمان، لا معلومات فوترة، لا رقم هاتف. المفتاح يبدأ بـ `gsk_`.

### 2. إعداد المفتاح

```bash
# الخيار A: ملف .env (موصى به)
echo 'GROQ_API_KEY=gsk_your_key_here' > .env

# الخيار B: متغير بيئة
export GROQ_API_KEY="gsk_your_key_here"
```

### 3. البرنامج يلتقطه

```python
# groq_provider.py
from dotenv import load_dotenv
load_dotenv()  # يحمّل .env في os.environ

api_key = os.getenv("GROQ_API_KEY")
if not api_key:
    api_key = os.environ.get("GROQ_API_KEY")
if not api_key:
    raise RuntimeError("لم يتم العثور على GROQ_API_KEY.")
```

ملف `.env` يُحمّل قبل التحقق من `os.environ` لذلك تعمل كلتا الطريقتين. الفحص المزدوج يعالج حالة عدم وجود `.env` مع استخدام `export`.

### التعليمات النظامية مضبوطة بدقة

**ما تفعله:**
- تحدد الدور: `"خبير macOS"` — تهيئ النموذج للتعرف على أسماء الخدمات
- تحدد التنسيق: `"5-8 كلمات"` — تفرض قيد عرض الطرفية
- تزيل الضوضاء: `"لا ماركداون، لا اقتباسات، لا علامات ترقيم"` — تضمن مخرجات نظيفة

**لماذا درجة الحرارة 0.2؟** درجات الحرارة الأعلى تجعل النموذج ينتج أوصافاً مختلفة لنفس العملية في استدعاءات مختلفة. عند 0.2، `WindowServer` يولّد دائماً شيئاً قريباً من "يدير عرض واجهة المستخدم الرسومية لـ macOS" — وهو ما نريده لسلوك حتمي ومخزّن.

**المعالجة البعدية:**
```python
desc = response.choices[0].message.content.strip().rstrip(".")
```
تزيل النقاط اللاحقة لأن التعليمات النظامية تقول بالفعل "لا علامات ترقيم في النهاية" — هذا يعالج الحالات التي يتجاهل فيها النموذج التعليمات.

## تحديد المعدل

المستوى المجاني لـ Groq يسمح بـ **30 طلباً في الدقيقة** و **14,400 طلباً في اليوم** على مستوى المؤسسة. البرنامج يطبق محدد معدل خاص به:

```python
class GroqProvider:
    def __init__(self):
        self._lock = threading.Lock()
        self._last_call = 0.0
        self.daily_count = 0
        self.max_daily = 14000  # هامش أمان أقل من 14,400

    def describe(self, name):
        with self._lock:
            # 1. فرض تأخير أدنى بين الاستدعاءات
            elapsed = time.time() - self._last_call
            if elapsed < Config.GROQ_DELAY_BETWEEN:  # 0.4 ثانية
                time.sleep(Config.GROQ_DELAY_BETWEEN - elapsed)

            # 2. التحقق من الميزانية اليومية
            if self.daily_count >= self.max_daily:
                return None

            self._last_call = time.time()
            self.daily_count += 1

        # 3. استدعاء API (خارج القفل)
        try:
            response = self.client.chat.completions.create(...)
            return response.choices[0].message.content.strip()
        except Exception:
            return None
```

**قرارات التصميم:**

- **نطاق القفل**: القفل يحمي فقط المؤقت والعداد. طلب HTTP الفعلي (`client.chat.completions.create`) يحدث **خارج** القفل حتى لا تحجب الخيوط المتعددة بعضها البعض أثناء I/O الشبكة.

- **هامش الأمان اليومي**: الحد الأقصى مضبوط على 14,000، تاركاً 400 طلب كمساحة رأس تحت حد 14,400. هذا يمنع فشل حافة الحد.

- **الفشل الصامت**: إذا أخطأ API (مشكلة شبكة، حد معدل، تحميل نموذج)، `describe()` يعيد `None`. المُستدعي في `background_resolver()` يمر إلى الاحتياط العام. البرنامج لا يتعطل أبداً بسبب خطأ API.

## التزامن

```python
# في config.py
GROQ_MAX_CONCURRENT = 5
GROQ_DELAY_BETWEEN = 0.4   # ثوانٍ
```

الحل الخلفي يعالج العناصر المعلّقة **تسلسلياً** (وليس بالتوازي) لأن القفل يسلسل استدعاءات API. مع `GROQ_DELAY_BETWEEN=0.4`، البرنامج يقوم بـ `60/0.4 = 150` طلباً في الدقيقة — وهو ما يتجاوز حد Groq البالغ 30 RPM لـ `llama-3.1-8b-instant`.

**هذا يعني أن تحديد المعدل يعتمد على تطبيق Groq من جانب الخادم**. إذا أعاد Groq 429 (تحديد معدل)، فإن `describe()` يلتقط الاستثناء ويعيد `None`، مما يسبب احتياطاً. هذا مقبول لأن:
1. المستخدم لن يلاحظ بعض العمليات التي ترجع إلى أوصاف عامة
2. الحد اليومي (14,000) هو القيد الحقيقي لمعظم حالات الاستخدام
3. زيادة التأخير إلى ثانيتين ستجعل الحل الخلفي أبطأ 5 مرات

للامتثال الصارم، اضبط `GROQ_DELAY_BETWEEN = 2.0` في `config.py`.

## استخدام الموارد

وصف واحد يكلف تقريباً:
- **الإدخال**: ~30 رمزاً (تعليمات نظامية + اسم عملية)
- **الإخراج**: ~10 رموز (وصف من 8 كلمات)
- **النطاق الترددي**: ~1 كيلوبايت لكل طلب
- **الوقت**: ~400-800 مللي ثانية عبر اتصال إنترنت نموذجي

لـ 350 عملية معلّقة:
- إجمالي الرموز: 350 × 40 = 14,000 رمزاً
- إجمالي الوقت: 350 × 0.5ث = ~3 دقائق
- إجمالي التكلفة: **0.00 دولار**

## مصفوفة معالجة الأخطاء

| نوع الخطأ | ماذا يحدث | تأثير المستخدم |
|-----------|-------------|-------------|
| لا مفتاح API | `GroqProvider()` يرفع `RuntimeError` | يتراجع إلى الأوصاف المحلية فقط |
| انتهاء مهلة الشبكة | يُلتقط الاستثناء → يُعيد `None` | العملية تحصل على وصف عام |
| تحديد المعدل (429) | يُلتقط الاستثناء → يُعيد `None` | بعض العمليات تحصل على أوصاف عامة |
| الوصول للحصة اليومية | فحص العداد → يُعيد `None` | العمليات المتبقية تحصل على أوصاف عامة |
| تحميل النموذج (503) | يُلتقط الاستثناء → يُعيد `None` | العملية تحصل على وصف عام |
| رد فارغ | `strip()` يعيد فارغاً → يُحفظ كما هو | يُظهر وصفاً فارغاً |
