### رموز التحكم ANSI

تستخدم الواجهة تسلسلين للهروب ANSI لتجنب وميض الشاشة:

```python
@staticmethod
def clear():
    sys.stdout.write("\033[2J\033[H")  # مسح الشاشة بالكامل + تحريك المؤشر للمنزل
    sys.stdout.flush()

@staticmethod
def cursor_home():
    sys.stdout.write("\033[H")          # تحريك المؤشر للمنزل WITHOUT مسح
    sys.stdout.flush()
```

- `\033[2J` — مسح العرض بالكامل
- `\033[H` — تحريك المؤشر للصف 0، العمود 0 (الموضع الرئيسي)

دالة `render()` تستدعي `cursor_home()` في البداية، ثم تكتب فوق المحتوى السابق سطراً بسطر. بما أن المخرجات الجديدة لها نفس ارتفاع المخرجات القديمة، يكون التأثير تحديثاً سلساً في المكان مع **صفر وميض**.

هذا يستبدل النهج الشائع لكن المشكل:
```python
os.system('clear')  # سيئ: يولد عملية فرعية، يسبب وميضاً مرئياً
```

### التصنيف والترتيب

```python
user = sorted(
    [p for p in snapshot if p.get("category") == "USER_APP"],
    key=lambda x: x["cpu"], reverse=True
)
```

كل فئة يتم ترشيحها وترتيبها حسب استخدام CPU تنازلياً. يتم عرض أفضل 50 عملية لكل فئة (قابل للتعديل عبر `SNAPSHOT_MAX_PER_CATEGORY`).

هذا يعني أن العمليات الأكثر استهلاكاً للموارد تظهر أولاً، وهو بالضبط ما يهتم به المستخدم.

### المعرفات القابلة للقتل

```python
killable = {}
idx = 1
for p in user:
    id_str = f"[{idx}]"
    killable[str(idx)] = (p["pid"], p["name"])
    idx += 1
```

فقط إدخالات `USER_APP` و `BACKGROUND_SERVICE` تحصل على معرفات قابلة للقتل. إدخالات `SYSTEM_CORE` تظهر `[-]` بدلاً من ذلك. قاموس `killable` يربط المعرفات النصية بزوج `(PID, name)` لمعالج الإدخال.

### شريط الحالة

```python
if resolver_active:
    print(" ⏳ جارٍ حل الأوصاف في الخلفية (Groq AI) ... 358 متبقياً")
    print("    [Q] خروج  |  [اكتب رقم المعرف] لقتل العملية (قتل فوري)")
else:
    print(" ✅ تم حل جميع الأوصاف.")
    print("    [Q] خروج  |  [R] تحديث  |  [اكتب رقم المعرف] لقتل العملية")
```

شريط الحالة يتغير حسب ما إذا كان الحل الخلفي لا يزال يعمل. لاحظ أن `[R] تحديث` يظهر فقط عند اكتمال الحل — التحديث أثناء الحل سيعيد تشغيل العملية ويضيع استدعاءات API.

### تخطيط الإدخال

```python
sys.stdout.write("الإجراء >> ")
sys.stdout.flush()
```

الموجه يجلس على سطره الخاص، جاهز لإدخال لوحة المفاتيح. `flush()` ضروري لأن `stdout` مخزن مؤقتاً سطرياً والموجه لا ينتهي بسطر جديد.

## معالجة الإدخال — حلقة `main()`

### قراءة غير حاجزة مع `select.select()`

```python
i, _, _ = select.select([sys.stdin], [], [], 0.3)
if i:
    user_input = sys.stdin.readline().strip().lower()
```

`input()` في Python تحجب إلى أجل غير مسمى، مما سيجمد الواجهة. بدلاً من ذلك، `select.select()` تتحقق مما إذا كان stdin لديه بيانات متاحة خلال 0.3 ثانية. إذا لم يصل إدخال:
- تعود الحلقة إلى `select.select()` لانتظار 0.3 ثانية أخرى
- الخيط الخلفي لا يزال يستطيع تحديث الأوصاف عبر استدعاءات `render()` الخاصة به
- الطرفية "تتوقف" بشكل فعال بينما تمتلئ الأوصاف

إذا وصل إدخال:
- `sys.stdin.readline()` تقرأه بدون حجب
- يُعالج الإدخال: `q` ← خروج، `r` ← تحديث، رقم ← قتل

### تدفق القتل

```python
if user_input in killable:
    pid, name = killable[user_input]
    print(f"\n⚠ إنهاء {name} (PID: {pid})؟ (y/n): ", end="", flush=True)
    confirm = sys.stdin.readline().strip().lower()
    if confirm == "y":
        try:
            os.kill(int(pid), 9)
            print(f"✅ تم إنهاء {name}.")
        except Exception as e:
            print(f"❌ فشل: {e}")
```

تدفق القتل:
1. ابحث عن المعرف المُدخل في قاموس `killable`
2. اطلب التأكيد (أمان النقر المزدوج)
3. أرسل `SIGKILL` (الإشارة 9) — إنهاء فوري
4. أعد عرض الجدول (العملية المقتولة تبقى مدرجة لأنها لقطة ثابتة)

**لماذا SIGKILL وليس SIGTERM؟** SIGTERM (الإشارة 15) يطلب من العملية إنهاء نفسها بشكل منظم. لمراقب العمليات، عندما يريد المستخدم قتل شيء ما، يريده عادةً أن يختفي فوراً. SIGKILL لا يمكن تجاهله أو معالجته — العملية تُنهى فوراً.

**معالجة الصلاحيات**: إذا كانت العملية تخص مستخدم آخر أو محمية بواسطة SIP (حماية تكامل النظام)، `os.kill()` يرفع `PermissionError`، الذي يُلتقط ويُعرض.

### تدفق التحديث

```python
if user_input == "r" and not resolver_active:
    TerminalUI.clear()
    TerminalUI.render(snapshot, killable, resolver_active=False)
```

التحديث يعيد عرض اللقطة الحالية. هو **لا** يعيد تشغيل `ps`. للحصول على قائمة عمليات جديدة، يجب على المستخدم الخروج وإعادة التشغيل.
