# نظرة عامة على البنية المعمارية

يستخدم Mac Monitor بنية **حل ثنائي المرحلة**: اعرض كل شيء فوراً، ثم أثرِه في الخلفية. هذا يضمن أن المستخدم لا ينتظر أبداً لكي يعرض الطرفية.

## المبدأ الأساسي

> المرحلة 1 (فورية):      لقطة ← تصنيف ← حل (ذاكرة/ثوابت/مسار) ← عرض
> المرحلة 2 (خلفية):      لكل عملية غير محلولة ← Whatis ← Groq AI ← ذاكرة ← إعادة عرض

يرى المستخدم جدولاً بالعمليات خلال **أجزاء من الثانية** من التشغيل. الأوصاف التي تظهر عليها علامة "⏳ جارٍ الحل..." تُملأ مباشرةً أثناء عمل العامل الخلفي.

## رسم بياني لتبعية الوحدات

```
mac_monitor_groq.py  (نقطة الدخول، المنسق)
     │
     ├── Config              (الإعدادات، التجاوزات، التصنيفات المعروفة)
     ├── ProcessSnapshot     (غلاف لأمر ps)
     ├── ProcessClassifier   (USER_APP / BACKGROUND / SYSTEM_CORE)
     ├── ProcessCache        (قاعدة بيانات SQLite)
     ├── DescriptionResolver (منسق سلسلة الاحتياط)
     ├── GroqProvider        (عميل AI مع محدد معدل)
     └── TerminalUI          (عرض ANSI، معالجة الإدخال)
```

التدفقات تكون **باتجاه واحد** — لا يوجد استيراد دائري. كل وحدة تستورد فقط ما تحتاجه من الوحدات المجاورة عبر `__init__.py`.

## تدفق البيانات

### تسلسل بدء التشغيل

```
1. ProcessSnapshot.capture()
   → يشغّل `ps -A -o pid,%cpu,%mem,comm`
   → يعيد list[dict] مع pid, cpu, mem, cmd, name

2. ProcessClassifier.classify(name, cmd)
   → فحص قائم على قواعد (KNOWN_CLASSES, .app/, com.apple., لاحقة d, ...)
   → يعيد "USER_APP" | "BACKGROUND_SERVICE" | "SYSTEM_CORE"

3. DescriptionResolver.resolve_immediate(name, cmd, category)
   → ذاكرة SQLite؟ أعِد
   → تجاوز ثابت؟ أعِد
   → استخراج من المسار؟ أعِد
   → أعِد None (أجّله للخلفية)

4. TerminalUI.render(...)
   → ترتيب حسب CPU تنازلياً داخل كل تصنيف
   → طباعة جدول ANSI مع أقسام ملونة
   → وسم الإدخالات غير المحلولة بـ "⏳ جارٍ الحل..."

5. إذا يوجد معلّق:
     GroqProvider()  → تحميل .env، التحقق من صحة مفتاح API
     threading.Thread(target=background_resolver).start()

6. while True:
     select.select([sys.stdin], timeout=0.3s)
     → معالجة 'q', 'r', قتل برقم
```

### حلقة الحل في الخلفية

```
لكل عملية معلّقة p:
   1. إذا p ليست USER_APP:
        desc ← whatis(p.name)             # محلي، بطيء (~2-10ث)
   2. إذا لا يزال لا يوجد desc:
        desc ← GroqProvider.describe(p.name)  # سحابي، سريع (~0.5ث)
   3. إذا لا يزال لا يوجد desc:
        desc ← f"عملية macOS: {p.name}"      # احتياط عام

   cache.set(p.name, desc, p.category, source)
   p.desc = desc
   p.resolved = True
   TerminalUI.render(...)  # إعادة عرض بالوصف المحدّث
```

## نموذج الخيوط

| الخيط | الدور | العمر |
|--------|------|----------|
| **الخيط الرئيسي** | يلتقط اللقطة، يعرض الواجهة، يعالج إدخال المستخدم (q, kill, refresh) | الجلسة بأكملها |
| **الحل الخلفي** (daemon) | يحلّق على القائمة المعلّقة، يستدعي whatis/Groq، يحدّث اللقطة، يعيد العرض | حتى يتم حل جميع الأوصاف |

خيط الخلفية هو **خيط خفي** — يموت تلقائياً عند خروج الخيط الرئيسي. لا يوجد تجمع خيوط معقد أو آليات غير متزامنة؛ مجرد حلقة خطية بسيطة فوق نسخة مستعارة من القائمة المعلّقة.

## قرارات التصميم الرئيسية

### 1. لماذا لا asyncio؟
يضيف `asyncio` في Python تعقيداً (إدارة حلقة الأحداث، برامج تشغيل قاعدة بيانات غير متزامنة). خيط خفي واحد مع I/O حاجز أبسط ويحقق نفس تجربة المستخدم: تظل الواجهة مستجيبة عبر `select.select()` بفاصل 0.3 ثانية.

### 2. لماذا لقطة ثابتة بدلاً من تحديثات حية؟
في الأصل، كان البرنامج يعيد تشغيل `ps` كل دورة، مما تسبب بمشكلتين:
- **الوميض**: `os.system('clear')` يخلق ومضات مرئية
- **القفز في الصفوف**: إعادة ترتيب العمليات حسب CPU كل دورة يجعل من المستحيل تتبع PID معين

نهج اللقطة الثابتة (التقاط مرة واحدة، عرض للأبد، تحديث الأوصاف فقط) حلّ كلتا المشكلتين.

### 3. لماذا حل خلفي دون ضمان الإكمال؟
الخيط الخفي هو إطلاق ونسيان. إذا قام المستخدم بقتل عملية أو خرج قبل وصول الأوصاف، لا شيء ينكسر. الذاكرة المؤقتة تحفظ النتائج للجلسة التالية.
