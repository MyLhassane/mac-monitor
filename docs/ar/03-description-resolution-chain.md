# سلسلة الاحتياط لحل الأوصاف

التحدي الأساسي الذي يحله هذا البرنامج: **كيف نقدم وصفاً واضحاً باللغة العربية لكل عملية تعمل على macOS، بدون أي جهد إعدادي وموثوقية عالية.**

الحل هو سلسلة احتياط من 6 مستويات. كل مستوى هو "محرك" مختلف بمقاييس مختلفة:

| المستوى | المحرك | السرعة | شبكة | الجودة | نسبة النجاح (تقريباً) |
|-------|--------|-------|---------|---------|-----------------|
| 1 | ذاكرة SQLite | <1ms | لا | يحافظ على أفضل ما هو متاح | 80%+ في التشغيل الثاني |
| 2 | تجاوزات ثابتة | <1ms | لا | منسقة يدوياً | ~2% |
| 3 | استخراج من المسار | <1ms | لا | غامض لكن دقيق | ~20% |
| 4 | Whatis | 2-10ث | لا | دقيق للخدمات | ~15% |
| 5 | Groq AI | 0.5ث | نعم | جودة عالية | ~60% |
| 6 | احتياط عام | <1ms | لا | "عملية macOS: {name}" | ~3% |

يتم تقييم السلسلة في `DescriptionResolver` (`modules/descriptions.py`). المستويات 1-3 **فورية** (حاجزة، تُستدعى من الخيط الرئيسي قبل أول عرض). المستويات 4-6 **خلفية** (تُستدعى من الخيط الخفي).

## المستوى 1: ذاكرة SQLite

```python
cached = self.cache.get(name)
if cached:
    return cached["description"], category, cached["source"]
```

في كل تشغيل، يتحقق البرنامج أولاً من `~/.mac-monitor-groq/process_cache.db`. إذا تم وصف عملية في جلسة سابقة، يُعاد الوصف فوراً بغض النظر عن المحرك الذي أنتجه.

**لماذا نخزّن كل شيء، وليس فقط ردود AI؟**  
لأن whatis بطيء (2-10 ثوانٍ لكل استدعاء). بمجرد الحصول على نتيجة whatis، تخزينها يوفر 2-10 ثوانٍ في كل تشغيل لاحق.

**إبطال الذاكرة**: الذاكرة **للقراءة فقط**. لا يوجد TTL أو إزالة. المنطق: أسماء العمليات لا تتغير (لن تقوم خدمة macOS اسمها `configd` فجأة بعمل شيء مختلف). إذا أردت مسح الذاكرة، احذف ملف قاعدة البيانات.

## المستوى 2: التجاوزات الثابتة

```python
if name in Config.HARDCODED_OVERRIDES:
    desc = Config.HARDCODED_OVERRIDES[name]
    self.cache.set(name, desc, category, "hardcoded")
    return desc, category, "hardcoded"
```

قاموس صغير في `config.py` يصلح **هلوسات** AI المعروفة. أثناء الاختبار مع Groq، بعض أسماء العمليات أنتجت أوصافاً خاطئة باستمرار:

| اسم العملية | Groq أعطى | التجاوز |
|-------------|-----------|----------|
| `Code Helper` | "مساعد تصحيح Xcode" | "عملية مضيف إضافات VS Code" |
| `Code Helper (Plugin)` | "إضافة Xcode للمساعدة البرمجية" | "عملية منفذ إضافات VS Code" |
| `Code Helper (Renderer)` | "يعرض المساعدة البرمجية والتمييز النحوي" | "عملية عرض VS Code" |
| `context7-mcp` | "معالجة الصوت الأساسية" | "خادم MCP موفر للسياق AI" |
| `opencode` | "يفتح محرر الكود المصدري" | "مساعد برمجة CLI مدعوم بـ AI" |

**متى تضيف تجاوزاً**: إذا لاحظت أن نفس العملية تحصل على وصف خاطئ في كل مرة، أضفها إلى `Config.HARDCODED_OVERRIDES`. المفتاح هو اسم العملية كما يظهر في عمود `Name` في الشاشة.

## المستوى 3: استخراج من المسار

```python
def _extract_from_path(self, cmd, name):
    m = re.search(r"/([^/]+)\.app/", cmd)
    if m:
        return f"مكون نشط من {m.group(1)}"
    if cmd.startswith(("/System/", "/usr/")):
        return None  # عملية نظام، أجلها إلى whatis/Groq
    return None
```

عندما تكون العملية داخل حزمة `.app`، يمكننا استخراج اسم التطبيق من مسارها:

| مسار الأمر | الوصف المستخرج |
|-------------|----------------------|
| `/Applications/Visual Studio Code.app/...` | "مكون نشط من Visual Studio Code" |
| `/Applications/Brave Browser.app/...` | "مكون نشط من Brave Browser" |
| `/System/Library/CoreServices/Finder.app/...` | "مكون نشط من Finder" |

هذا ينتج أوصافاً **غامضة لكنها ليست خاطئة أبداً**. إنها احتياط، وليست النتيجة المثالية. الخيط الخلفي سيستبدلها لاحقاً بوصف AI أفضل.

**لماذا لا نستخرج من Info.plist؟** إصدار سابق استخدم `plistlib` لتحليل `Info.plist` لاستخراج `CFBundleDisplayName`. تم التخلي عن هذا لأن:
1. بطيء (إدخال/إخراج قرص لكل عملية)
2. العديد من حزم `.app` هي أطر متداخلة بدون Info.plist خاص بها
3. نهج التعبير المنتظم فوري وجيد كاحتياط

## المستوى 4: Whatis (خلفية)

```python
def _try_whatis(self, name):
    clean = re.sub(r"\(.*?\)", "", name).split()[0]
    try:
        out = subprocess.check_output(f"whatis {clean} 2>/dev/null", ...)
        for line in out.splitlines():
            if line.strip().startswith(clean + "(") or line.strip().startswith(clean + " "):
                parts = re.split(r"\s+[–-]\s+", line, maxsplit=1)
                return parts[1].strip()[:80]
    except:
        return None
```

`whatis` يبحث في قاعدة بيانات الصفحات اليدوية للنظام (mandb) ويعيد أوصافاً من سطر واحد. إنه نفس الأمر الذي يشغّل `man -f`.

**لماذا هو في مسار الخلفية**: على macOS، `whatis` بطيء بشكل مفاجئ:
- الاستدعاء الأول: ~2-5 ثوانٍ (أحياناً يشغّل إعادة بناء `makewhatis`)
- الاستدعاءات التالية: ~1-3 ثوانٍ لكل عملية
- أسوأ حالة (نتائج كثيرة مثل `launchd`): 8+ ثوانٍ

إذا تم استدعاؤه بشكل متزامن لـ 100 خدمة، سينتظر المستخدم **دقائق** قبل رؤية أي شيء.

**تفاصيل المحلل**: مخرجات `whatis launchd` الخام تبدو كالتالي:
```
WiFiVelocityAgent(8) - وكيل launchd لإطار WiFiVelocity
launchctl(1)         - يتفاعل مع launchd
launchd(8)           - مدير الخدمات على مستوى النظام والمستخدم   ← هذه
launchd.plist(5)     - ملفات إعدادات الخدمات على مستوى النظام والمستخدم
```

يجد المحلل السطر الذي يظهر فيه اسم الأمر قبل `(` مباشرة ويستخرج النص بعد ` - `.

**Whatis يُجرّب فقط للعمليات غير USER_APP**. إذا كان المصنف قد حدد بالفعل أن العملية هي تطبيق مستخدم، لا فائدة من سؤال whatis (لن يكون لديه صفحة دليل).

## المستوى 5: Groq AI (خلفية)

```python
desc = groq.describe(p["name"])
```

إذا لم يُرجع whatis شيئاً، يسأل البرنامج ذكاء اصطناعياً سحابياً مجانياً (Groq). انظر [04-groq-ai-integration.md](../en/04-groq-ai-integration.md) للتفاصيل الكاملة.

المعلمات الرئيسية:
- النموذج: `llama-3.1-8b-instant` (الأسرع، 14,400 طلب/يوم مجاناً)
- التعليمات النظامية: `"أنت خبير macOS. قدم وصفاً تقنياً قصيراً (5-8 كلمات) لاسم عملية macOS المحدد."`
- درجة الحرارة: 0.2 (إبداع منخفض، مخرجات متناسقة)
- الحد الأقصى للرموز: 30

الذكاء الاصطناعي محسّن لـ **الإيجاز والتناسق** بدلاً من التحليل العميق. الرد النموذجي هو:

```
WindowServer → "يدير عرض واجهة المستخدم الرسومية لـ macOS"
configd     → "مدير إعدادات النظام والشبكة"
Python      → "لغة برمجة عالية المستوى ومفسّرة"
```

## المستوى 6: الاحتياط العام

```python
if not desc:
    desc = f"عملية macOS: {p['name']}"
```

إذا فشلت كل المحركات (فقدان الذاكرة، لا تجاوز، لا مسار .app، whatis انتهت مهلة، Groq أخطأ)، تحصل العملية على وصف واقعي لكن غير مفيد. هذا نادر عملياً (<3% من العمليات).

## الفصل بين `resolve_immediate` و `resolve_background`

```python
# تُستدعى من الخيط الرئيسي — يجب أن تكون فورية
def resolve_immediate(self, name, cmd, category):
    cached = self.cache.get(name)
    if cached: return ...
    if name in HARDCODED_OVERRIDES: return ...
    path_desc = self._extract_from_path(cmd, name)
    if path_desc: return ...
    return None  # أجل للخلفية

# تُستدعى من الخيط الخلفي — يمكن أن تكون بطيئة
def resolve_background(self, name, cmd, description, category, source):
    self.cache.set(name, description, category, source)
```

التصميم ذو الأسلوبين يجعل المسار الحاجز/السريع واضحاً. `resolve_immediate` لا تستدعي whatis أو Groq أبداً — إنها strictly cache → overrides → path extraction. `resolve_background` هي غلاف بسيط حول `cache.set()`؛ منطق whatis/Groq الفعلي يعيش في `mac_monitor_groq.py:background_resolver()`.

## قياس نسبة النجاح

على Mac نموذجي بحوالي 480 عملية:

| المرحلة | تم الحل | % |

في **التشغيل الثاني**، تحوّل الذاكرة معظم حلول الخلفية إلى فورية:

| عمليات جديدة تحتاج خلفية | ~10 | 2% |
